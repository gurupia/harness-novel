## 1. 패러다임 전환: 3단계 진화

| 시기 | 패러다임 | 핵심 질문 | 비유 |
|------|---------|----------|------|
| 2022~2024 | **Prompt Engineering** | "어떻게 말해야 하나?" | 업무 지시서 |
| 2025 | **Context Engineering** | "모델이 지금 무엇을 알아야 하나?" | 브리핑 패킷 |
| 2026 | **Agent Harness Design** | "시스템이 어떻게 안전하고 안정적으로 작동하나?" | 운영체제(OS) |

### 핵심 공식

```
Agent = Model + Harness
```

- **Model** = 지능(추론)을 제공
- **Harness** = 제어(행동, 안전, 신뢰성)를 제공
- 2026년 기준 모델 간 성능 차이는 미미. **경쟁력의 원천은 하네스 품질**

## 2. 프로덕션 하네스의 4대 책임

| 책임 | 설명 |
|------|------|
| **도구 실행 & 권한** | 에이전트가 무엇을 할 수 있는지 제한. 도구 허용 목록, 역할 기반 접근 제어 |
| **메모리 & 상태 관리** | 장기 지속성, 세션 간 상태 유지, 계층형 메모리(단기/장기) |
| **가드레일 & 안전** | 경계 강제, 인간 개입 체크포인트, Pre/Post LLM 가드레일 |
| **관찰 가능성** | 에이전트 의사결정 추적, 도구 호출 결과 로깅, 실패 모드 디버깅 |

## 3. 핵심 아키텍처 패턴

### 3.1 Planner-Generator-Evaluator (PGE) 패턴

창작 AI 워크플로우의 **업계 표준 패턴**. GAN(적대적 생성 네트워크)에서 영감.

```
┌──────────┐     계획     ┌──────────┐    초안    ┌──────────┐
│ Planner  │ ──────────► │Generator │ ────────► │Evaluator │
│ (기획)   │              │ (집필)   │            │ (검증)   │
└──────────┘              └──────────┘            └──────────┘
                               ▲                       │
                               │   🔴 실패 시 피드백     │
                               └───────────────────────┘
```

#### 핵심 원칙
1. **컨텍스트 분리 (가장 중요):** Evaluator는 Generator와 반드시 컨텍스트를 분리해야 함. 같은 대화 히스토리를 공유하면 Evaluator가 Generator의 편향을 물려받아 오류를 발견하지 못함
2. **객관적 기준:** "잘 됐는지"를 주관이 아닌 체크리스트·루브릭으로 판정
3. **관리된 핸드오프:** 오케스트레이터(Harness)가 상태 관리, 피드백 라우팅, 경계 강제를 담당
4. **최대 반복 제한:** 무한 루프 방지. 보통 최대 2~3회 반복 후 인간에게 위임

#### 창작 분야 적용
| 역할 | 소설 집필에서의 실행 |
|------|-------------------|
| **Planner** | 비트시트, 인물노트, 감각팔레트, 대화설계 (4종 기획 세트) |
| **Generator** | 기획 산출물 기반 6,000~7,000자 원고 집필 |
| **Evaluator** | 5-PASS 검증 + 문장 폴리싱 + 설정 대조 |

### 3.2 Supervisor (Orchestrator-Worker) 패턴

중앙 오케스트레이터가 복잡한 태스크를 분해하여 전문 에이전트에 할당.

- **적합 상황:** 명확한 태스크 경계, 결정론적 실행이 필요한 경우
- **약점:** 오케스트레이터가 단일 장애점(SPOF)이 될 수 있음

### 3.3 Split-and-Merge (병렬 팬아웃) 패턴

코디네이터가 태스크를 병렬 서브태스크로 분할, 동시 실행 후 결과 병합.

- **적합 상황:** 독립적인 하위 문제 처리, 속도가 중요한 경우
- **소설 적용 예:** 여러 회차 동시 기획, 설정 일관성 병렬 검증

## 4. 가드레일 설계 패턴

### Pre-LLM 가드레일 (입력 전)

| 유형 | 설명 |
|------|------|
| **정보 위생** | PII 제거, 민감 데이터 차단, 프롬프트 인젝션 탐지 |
| **입력 필터링** | 범위 밖 요청 차단, 유해성 차단 |
| **컨텍스트 주입** | 필요한 설정·상태만 선별적으로 컨텍스트에 주입 |

### Post-LLM 가드레일 (출력 후)

| 유형 | 설명 |
|------|------|
| **자기 교정 루프** | 출력이 체크 실패 시, 타겟 교정 프롬프트로 재시도 |
| **신뢰도 게이트** | 신뢰도 점수가 임계값 이하면 차단 또는 인간 개입 |
| **둠 루프 탐지** | 연속 동일 도구 호출 핑거프린팅으로 무한 루프 방지 |

### 창작 분야 가드레일

| 기법 | 설명 |
|------|------|
| **Critic-Generator 패턴** | 생성 LLM과 별도의 평가 LLM(또는 규칙 기반 검증기)이 브랜드/스타일 가이드라인 대조 |
| **Blueprint 구조 강제** | 하네스가 outline → draft → polish 단계를 순차 강제 |
| **클리셰 필터링** | 과다 사용 표현을 키워드/의미 모니터링으로 탐지 후 재작성 강제 |

## 5. Subtraction Architecture (뺄셈 아키텍처)

> **핵심 원칙:** 도구나 역할을 추가하는 것보다, 불필요한 것을 제거하면 성공률이 올라간다.

### 적용 지침

1. **중복 도구 제거:** 같은 기능을 하는 도구가 2개면 1개로 통합
2. **의사결정 트리 단순화:** 에이전트가 선택할 도구/경로가 적을수록 정확도 상승
3. **최소 도구 원칙:** 시작은 2~3개 전문 에이전트 + 명확한 오케스트레이션. 복잡도는 점진 추가
4. **컨텍스트 예산 관리:** 컨텍스트 윈도우는 희소 자원. 무제한 메모리가 아님

### 컨텍스트 관리 전략

| 컨텍스트 유형 | 설명 | 관리 방법 |
|-------------|------|----------|
| **정적 컨텍스트** | 시스템 지침, 행동 규칙, 도구 정의 | 캐싱 가능, 한 번 로드. 압축 대상 |
| **동적 컨텍스트** | 대화 이력, 도구 출력, 태스크 상태 | 엄격 관리. 오래된 이력 요약·폐기 |

### 프로덕션 원칙

- **결정론적 "실행" vs 확률적 "사고":** LLM은 추론과 의도 추출에만 사용. 실제 계산·데이터 기록은 결정론적 함수(스크립트)가 처리
- **실패를 전제한 설계:** 개별 에이전트는 실패할 수 있다고 가정. 재시도, 상태 롤백, 명시적 실패 처리 경로를 하네스에 구현
- **결과 측정:** 토큰 수나 프롬프트 속도가 아닌, **태스크 완료율과 시스템 신뢰성**을 추적

## 6. 멀티 에이전트 오케스트레이션 원칙

### 시작은 모놀리스로

> 2025년 업계 교훈: **단일 에이전트 루프가 특정 컨텍스트·도구·추론 한계에 도달했을 때만** 멀티 에이전트로 전환.

### 실행 로드맵

1. **신뢰성에 집중:** 모델/프롬프트 최적화가 아니라, 가드레일과 계측에 투자
2. **단순하게 시작:** 2~3개 전문 에이전트 + 명확한 오케스트레이션 패턴. 복잡도는 점진 추가
3. **결과 측정:** 태스크 완료율과 시스템 신뢰성 추적
4. **실패 설계:** 개별 에이전트 실패를 가정. 재시도 로직, 상태 롤백, 명시적 실패 처리 경로 구현

### 역할 분리 원칙

| 원칙 | 설명 |
|------|------|
| **전문화** | 각 에이전트는 자기 전문 영역에서만 의견. Editor가 플롯을 바꾸지 않음 |
| **건설적 비판** | "안 된다"가 아니라 "이렇게 하면 더 좋다"로 |
| **근거 제시** | 모든 제안에 장르 문법·독자 기대·서사 논리 등 이유 명시 |
| **사용자 주권** | 팀 의견 충돌 시 최종 결정은 사용자 |
