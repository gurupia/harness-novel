> **버전:** 3.0

> **참조:** `harness-engineering-reference.md`

## 1. 개요

Harness Layer는 범용 팀의 **모든 파이프라인 단계에 적용되는 운영 인프라**이다.  
에이전트가 아닌, 에이전트들이 작동하는 **환경과 규칙**을 정의한다.

```
┌──────────────────────────────────────────────┐
│              Harness Layer                   │
│  ┌────────┐  ┌────────┐  ┌────────┐          │
│  │ 상태    │  │ 오류   │  │ 비용    │          │
│  │ 관리    │  │ 복구   │  │ 추적    │          │
│  └────────┘  └────────┘  └────────┘          │
│         ▲          ▲          ▲              │
│         └──────────┴──────────┘              │
│              모든 에이전트 보고                 │
└──────────────────────────────────────────────┘
```

## 2. 상태 관리 (State Management)

### 2.1 파이프라인 상태 추적

각 파이프라인 단계의 상태를 추적한다:

| 상태 | 의미 |
|------|------|
| `PENDING` | 대기 중 (이전 단계 미완료) |
| `RUNNING` | 해당 에이전트 실행 중 |
| `DONE` | 완료, 다음 단계로 전달 가능 |
| `FAILED` | 실패, 오류 복구 필요 |
| `BLOCKED` | 사용자 입력 대기 |

**상태 로그 형식:**
```markdown
## 파이프라인 상태 — 제{N}화
| 레이어 | 단계 | 에이전트 | 상태 | 비고 |
|--------|------|---------|------|------|
| Planner | 1 | Plotter | DONE | 비트시트 생성 완료 |
| Planner | 2 | Character | DONE | 인물노트 + 대화설계 완료 |
| Planner | 3 | Stylist | DONE | 감각팔레트 완료 |
| Generator | 4 | Writer | RUNNING | 원고 집필 중 |
| Evaluator | 5 | Reviewer | PENDING | — |
| Evaluator | 6 | Editor | PENDING | — |
```

### 2.2 컨텍스트 보존

세션 간 컨텍스트를 보존하기 위한 핸드오프 프로토콜:

```markdown
## 세션 핸드오프 — {YYYY-MM-DD HH:MM}

### 완료 작업
- {완료 사항 목록}

### 다음 세션 예정
- {예정 작업}

### 활성 상태
- 파이프라인 위치: {어느 단계까지 완료}
- 미해결 이슈: {있으면 기술}
- 장르 모듈 활성: {사용 중인 모듈 목록}

### 인물 아크 현재 위치
- {인물}: {아크 단계}

### 미회수 떡밥
- {떡밥}: {상태}
```

## 3. 오류 복구 (Error Recovery)

### 3.1 오류 유형 및 대응

| 오류 유형 | 증상 | 발생 레이어 | 자동 복구 | 수동 복구 |
|----------|------|-----------|---------|----------|
| **분량 미달** | 5,500자 미만 | Evaluator | Generator에게 분량 확보 지시 (감각 레이어링 우선) | 사용자에게 보강 방향 확인 |
| **설정 모순** | Reviewer PASS 3 실패 | Evaluator | Generator에게 해당 구간 표시 후 반려 | 사용자에게 어느 쪽이 정본인지 확인 |
| **톤 이탈** | Reviewer PASS 2 실패 | Evaluator | Generator에게 톤 재조정 피드백 | — |
| **플롯 누락** | 비트시트 사건 미반영 | Generator | Planner에게 비트시트 재설계 요청 | 사용자에게 방향 확인 |
| **장르 부정합** | 장르 모듈 검증 실패 | Evaluator | 해당 모듈 재호출 | 사용자에게 장르 방향 확인 |
| **기획 품질 부족** | 4종 세트 불완전 | Planner | 누락 항목 재생성 | — |

### 3.2 반복 한도

```
자동 복구 최대 횟수: 2회
2회 실패 후 → 사용자에게 판단 위임
위임 시 반드시 포함: 실패 원인 + 시도한 해결책 + 선택지 2~3개
```

### 3.3 에스컬레이션 경로

```
에이전트 자체 해결 → Harness 자동 복구 → 사용자 위임
     (1차)              (2차)              (최종)
```

## 4. 비용 추적 (Cost Tracking)

### 4.1 가상 비용 모델

실제 API 비용이 아닌, **작업 복잡도 기반 가상 비용**으로 추적한다.

| 작업 유형 | 가상 비용 단위 | 근거 |
|----------|-------------|------|
| 비트시트 생성 | 1.0 unit | 단일 구조 설계 |
| 인물 노트 | 0.8 unit | 기존 설정 기반 |
| 감각 팔레트 | 0.6 unit | 톤 매트릭스 참조 |
| 통합 집필 (6,000자) | 3.0 unit | 핵심 생산 단계 |
| 리뷰 (5-PASS) | 1.5 unit | 전수 검토 |
| 에디팅 | 1.0 unit | 문장 수준 수정 |
| 장르 모듈 호출 | 0.5 unit/회 | 부분 보강 |
| 수정 재작성 | 2.0 unit | 부분 재집필 |

**회차 1개 표준 비용:** ~8.0 unit (모듈 미사용 시)

### 4.2 비용 보고 형식

```markdown
## 비용 보고 — 제{N}화
| 레이어 | 단계 | 에이전트 | 비용 | 재시도 | 총계 |
|--------|------|---------|------|--------|------|
| Planner | 비트시트 | Plotter | 1.0 | 0 | 1.0 |
| Planner | 인물노트 | Character | 0.8 | 0 | 0.8 |
| Planner | 감각팔레트 | Stylist | 0.6 | 0 | 0.6 |
| Generator | 원고 집필 | Writer | 3.0 | 0 | 3.0 |
| Evaluator | 5-PASS 검증 | Reviewer | 1.5 | 0 | 1.5 |
| Evaluator | 폴리싱 | Editor | 1.0 | 0 | 1.0 |
| **합계** | — | — | — | — | **7.9** |
```

## 5. 멀티 프로젝트 관리

### 5.1 프로젝트 전환 프로토콜

| 상황 | 조치 |
|------|------|
| 활성 문서 경로에서 다른 프로젝트 감지 | "《{프로젝트명}》으로 전환하시겠습니까?" 확인 |
| 동일 세션 내 전환 | 이전 프로젝트 핸드오프 자동 생성 후 전환 |
| 프로젝트 식별 불가 | 사용자에게 명시적 확인 |

### 5.2 프로젝트 격리

- 각 프로젝트의 인물·세계관·떡밥 상태는 **독립적으로 관리**
- 프로젝트 간 설정 혼선 방지를 위해 전환 시 컨텍스트를 완전히 교체
